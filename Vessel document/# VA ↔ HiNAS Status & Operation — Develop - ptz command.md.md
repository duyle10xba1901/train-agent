# VA ↔ HiNAS Status & Operation — Developer Spec

**Date:** 2026-04-20
**Scope:** HiNAS-side (Navigation Autonomy) Status & Operation interfaces exchanged between Vessel Autonomy (VA) and HiNAS. VMS-side Status & Operation (PCS / PMS / Auxiliary / Damage Control) is **out of scope** for this document and remains TBD pending VMS (HiCONiS) interface finalisation.
**Classification:** CONFIDENTIAL

> **⚠ Perception Interface Instability (HiNAS Rev. 1.2.2+):** PTZ mode (`ptz_mode` in `$PAVKHOMS`, PTZ-specific indication items) and Radar mode (`radar_mode` in `$PAVKHOMS`, RADAR-specific indication items) are part of the Perception group that is being restructured. Their presence inside the HOMS sentence and the Indication snapshot is assumed stable for this document; downstream mode-specific command interfaces (PTZ, Radar) are covered in separate specs and are explicitly flagged unstable there.

---

## 1. Overview

"Status & Operation" on the Navigation Autonomy boundary is defined as:

- **Operation (command / set)** — VA → HiNAS mode commands that change the autonomy configuration (CAGA on/off, PTZ mode, Radar mode). One exception is the steering command `$--HTC`, which is **generated by HiNAS** and **relayed by VA** to the vessel autopilot.
- **Status (telemetry / report)** — HiNAS → VA periodic reports of the current autonomy mode, track-controller operational state, and per-mode health indications.

Common transport characteristics:

| Item                  | Value                                        |
| --------------------- | -------------------------------------------- |
| NMEA transport        | IEC 61162-450 Ed.3 **UdPbC** (UDP multicast) |
| JSON transport        | IEC 61162-450 Ed.3 **RaUdP**                 |
| Encoding              | UTF-8 (NMEA ASCII; JSON raw UTF-8)           |
| Connection monitoring | HBT every 0.5 s; 5 s silence = LINK LOST     |

Mode-specific commands for PTZ and Radar are **not duplicated** in this document. See:

- PTZ — `VA_HiNAS_PTZControl_Spec.md`
- Radar — `VA_HiNAS_RadarControl_Spec.md` (planned)
- Risk Assessment (adjacent status channel) — `VA_HiNAS_RiskAssessment_Spec.md`

## 3. Group A — Operation Mode Commands (VA → HiNAS)

### 3.1 CAGA Mode — `$PKSOCAGM` / `$PAVKCAGM`

Enables or disables CAGA (Collision Avoidance and Grounding Avoidance) operating mode. CAGA is orthogonal to the Mission RTZ flow: when CAGA is enabled, HiNAS autonomously emits Local Path Plan (LPP) updates for collision and grounding avoidance **without** changing `routeStatus`.

**Command (VA → HiNAS) — working format:**

```
$PKSOCAGM,{caga_mode},{status}*hh
```

**Response (HiNAS → VA) — working format:**

```
$PAVKCAGM,{caga_mode},{result},{status}*hh
```

| Field       | Values    | Description                                |
| ----------- | --------- | ------------------------------------------ |
| `caga_mode` | `E` / `D` | `E` = Enable, `D` = Disable                |
| `result`    | `A` / `R` | `A` = Accept, `R` = Reject (response only) |
| `status`    | `A` / `V` | `A` = Valid, `V` = Invalid                 |

### 3.2 PTZ Mode — covered in `VA_HiNAS_PTZControl_Spec.md`

PTZ mode command (`$PKSOPTZM`) and echo (`$PAVKPTZM`) are defined in the PTZ spec. The current PTZ mode is additionally reported inside `$PAVKHOMS.ptz_mode` (§5.1 below).

### 3.3 Radar Mode — covered in `VA_HiNAS_RadarControl_Spec.md` (planned)

Radar mode command (`$PKSORDRM`) and echo (`$PAVKRDRM`) are defined in the Radar spec. The current radar mode is additionally reported inside `$PAVKHOMS.radar_mode` (§5.1 below).

### 3.4 Command Timeout and Retry (applies to A1 / A3 / A4)

| Item                | Value                                                                   |
| ------------------- | ----------------------------------------------------------------------- |
| Echo wait timeout   | **1.0 s**                                                               |
| Retry count         | up to **3 retransmissions**                                             |
| Failure declaration | after 3 failed attempts, mark command as failed and surface to operator |

## 4. Group B — HTC Relay (HiNAS → VA → Autopilot)

HiNAS generates the steering command `$--HTC` (IEC 61162-1). VA is responsible for **relaying** this sentence to the vessel autopilot system (via Machinery Autonomy). VA does not originate HTC.

| Item      | Value                               |
| --------- | ----------------------------------- |
| Origin    | HiNAS                               |
| Direction | HiNAS → VA → Autopilot (relay)      |
| Format    | IEC 61162-1 HTC (standard sentence) |
| Period    | 0.5 s (2 Hz)                        |
| Transport | UdPbC                               |

**VA responsibilities:**

- Forward each received `$--HTC` to the vessel autopilot without modifying the talker ID.
- On HBT link loss, **cease forwarding** HTC to the autopilot (failsafe).
- If HiNAS does not receive VA's complementary `$--HTD` + `$PKSOHSTS` for **5 consecutive cycles (2.5 s at 2 Hz)**, HiNAS reverts steering mode to Standalone (`S`) and releases heading control.
- Heading control and speed control operate independently — loss of one does not disable the other.

> **CRITICAL:** VA must never inject, synthesise, or cache HTC commands. HTC is a live autopilot command generated by HiNAS; stale or fabricated HTC values can drive the vessel into unsafe heading states.

## 5. Group C — Status Telemetry (HiNAS → VA)

### 5.1 `$PAVKHOMS` — HiNAS Operation Mode Status

Periodic top-level autonomy mode status covering Track Control, Collision Avoidance, Radar, and PTZ.

**Format:**

```
$PAVKHOMS,{track_control_status},{collision_avoidance_status},{radar_mode},{ptz_mode}*hh
```

| Field                        | Values          | Description                                           |
| ---------------------------- | --------------- | ----------------------------------------------------- |
| `track_control_status`       | `R` / `N` / `A` | `R` = Ready, `N` = Not Ready, `A` = Activate          |
| `collision_avoidance_status` | `R` / `N` / `A` | `R` = Ready, `N` = Not Ready, `A` = Activate          |
| `radar_mode`                 | `B` / `A` / `T` | `B` = Bypass, `A` = Auto ARPA, `T` = Target Selection |
| `ptz_mode`                   | `M` / `A`       | `M` = Manual, `A` = Auto                              |

| Item      | Value            |
| --------- | ---------------- |
| Period    | 1.0 s (1 Hz)     |
| Transport | UdPbC (port TBD) |

Treat `$PAVKHOMS` as the **authoritative steady-state** for each of the four modes. The per-mode command echoes (`$PAVKCAGM`, `$PAVKPTZM`, `$PAVKRDRM`) are transactional acknowledgements only, not steady-state indicators.

### 5.2 `track_control_info` — Track Controller Operational Status

Current operational state of the active track controller, keyed to the active track leg (GPP or LPP).

**Transport:** JSON (UTF-8) over RaUdP, 0.5 s (2 Hz).

| Field                              | Type          | Unit                | Description                                               |
| ---------------------------------- | ------------- | ------------------- | --------------------------------------------------------- |
| `issued_time`                      | string        | --                  | ISO 8601 UTC timestamp                                    |
| `source`                           | string / null | --                  | `"GPP"` / `"LPP"`                                         |
| `heading_control_method`           | string / null | --                  | `"Straight"` / `"Turn"`                                   |
| `corridor_status`                  | string / null | --                  | `"InCorridor"` / `"OffCorridor"`                          |
| `track_position_status`            | string / null | --                  | e.g. `"OnTrack"`, `"EndOfTrack"`                          |
| `active_track_waypoint_number`     | int / null    | --                  | 0-999                                                     |
| `active_track_waypoint_name`       | string / null | --                  | Max 64 chars                                              |
| `active_track_bearing_deg`         | double / null | deg                 | 0.00-359.99                                               |
| `active_track_type`                | string / null | --                  | `"RL"` (Rhumb Line) / `"GC"` (Great Circle)               |
| `active_track_tangent_bearing_deg` | double / null | deg                 | Instantaneous tangential bearing                          |
| `active_track_sog`                 | double / null | kn                  | 0.00-99.99                                                |
| `active_track_rpm`                 | double / null | RPM                 | Planned RPM                                               |
| `commanded_rpm`                    | double / null | RPM                 | Currently commanded RPM                                   |
| `next_track_waypoint_name`         | string / null | --                  | Max 64 chars                                              |
| `next_track_bearing_deg`           | double / null | deg                 | 0.00-359.99                                               |
| `next_track_speed_kn`              | double / null | kn                  | 0.00-99.99                                                |
| `along_track_distance_nm`          | double / null | NM                  | Distance travelled along active leg (ATD)                 |
| `dwol_nm`                          | double / null | NM                  | Distance to Wheel-Over Line                               |
| `twol`                             | string / null | HH:MM:SS            | Time to Wheel-Over Line                                   |
| `eta`                              | string / null | YYYY-MM-DD HH:MM:SS | ETA at final destination                                  |
| `xte_nm`                           | double / null | NM                  | Cross Track Error (negative = port, positive = starboard) |

Null is allowed on any field when the corresponding measurement is not currently available.

### 5.3 Indication — Schema Exchange + State Snapshot

HiNAS publishes per-mode health / readiness information through an indication schema and a periodic all-mode snapshot. VA must request the schema once on connection and cache it to interpret the snapshots.

#### 5.3.1 `$PKSOINDR` — Indication Schema Request (VA → HiNAS)

**Format:**

```
$PKSOINDR,{version_major},{version_minor},{version_hotfix}*hh
```

| Field            | Values      | Description                                            |
| ---------------- | ----------- | ------------------------------------------------------ |
| `version_major`  | int / empty | Cached schema major version; empty on first connection |
| `version_minor`  | int / empty | Cached schema minor version                            |
| `version_hotfix` | int / empty | Cached schema hotfix version                           |

| Item      | Value                                                  |
| --------- | ------------------------------------------------------ |
| Trigger   | On connection, on version mismatch, after reconnection |
| Retry     | 3 s timeout, max 3 retries (4 attempts total)          |
| Transport | UdPbC                                                  |

#### 5.3.2 `nas_indication_schema` — Indication Schema Response (HiNAS → VA)

Raw UTF-8 JSON over RaUdP. Maximum payload 50 KB.

| Field                                      | Type   | Description                                   |
| ------------------------------------------ | ------ | --------------------------------------------- |
| `issued_time`                              | string | ISO 8601 UTC                                  |
| `version_major`                            | int    | Schema major version                          |
| `version_minor`                            | int    | Schema minor version                          |
| `version_hotfix`                           | int    | Schema hotfix version                         |
| `modes[]`                                  | array  | One entry per mode (TCS / CAGA / RADAR / PTZ) |
| `modes[].mode_name`                        | string | Mode identifier                               |
| `modes[].indication_items[]`               | array  | Item list for this mode                       |
| `modes[].indication_items[].indication_id` | int    | 1000-9999 (indication ID range per mode)      |
| `modes[].indication_items[].title`         | string | Max 50 chars                                  |
| `modes[].indication_items[].description`   | string | Max 300 chars                                 |

Indication ID range allocation (from VA convention):

| Range  | Mode                       |
| ------ | -------------------------- |
| `1xxx` | TCS (Track Control System) |
| `2xxx` | CAGA                       |
| `3xxx` | RADAR                      |
| `4xxx` | PTZ                        |

VA must cache the schema keyed by version; the snapshot (§5.3.3) references items only by `indication_id`.

#### 5.3.3 `$PAVKINDD` — All-Mode Indication Snapshot (HiNAS → VA)

All-mode / all-item state snapshot, replacing the legacy `$PAVKINDC` (cyclic) + `$PAVKINDE` (event) pair on the VA side.

**Format (conceptual — exact sentence layout pending alignment):**

```
$PAVKINDD,{sentence_total},{sentence_no},{message_id},{version_major},{version_minor},{version_hotfix},{mode},{id_1},{state_1},{id_2},{state_2},...*hh
```

| Field            | Values                           | Description                                                     |
| ---------------- | -------------------------------- | --------------------------------------------------------------- |
| `sentence_total` | int                              | Total number of sentences in this snapshot                      |
| `sentence_no`    | int                              | Current sentence index                                          |
| `message_id`     | int                              | Snapshot message ID                                             |
| `version_*`      | int                              | Schema version currently in effect                              |
| `mode`           | `TCS` / `CAGA` / `RADAR` / `PTZ` | Mode the following items belong to                              |
| `id_n`           | int                              | `indication_id` defined in the cached schema                    |
| `state_n`        | `N` / `D` / `F`                  | `N` = Normal, `D` = Degraded (replaces legacy `L`), `F` = Fault |

| Item           | Value                            |
| -------------- | -------------------------------- |
| Period         | 30 s + on state change           |
| Multi-sentence | Split at 82-character NMEA limit |
| Transport      | UdPbC                            |

> **NOTE:** If HiNAS publishes legacy `$PAVKINDC` / `$PAVKINDE` instead of the consolidated `$PAVKINDD`, VA must merge both into a single logical snapshot keyed by `(mode, indication_id)`. Alignment on the final sentence tag is pending with AVIKUS.

## 6. VA Implementation Responsibilities

### 6.1 Transmitters (VA → HiNAS)

1. **`$PKSOCAGM` (A1)** — Send on operator CAGA-toggle command only. Retry per §3.4.
2. **`$PKSOINDR` (C3)** — Send on connection establishment, on detected schema version mismatch, and on reconnection. Retry per §5.3.1.

### 6.2 Receivers (HiNAS → VA)

1. **`$PAVKCAGM` echo (A2)** — Match to the outstanding CAGM command; clear retry counter on valid `result = A`; on `result = R`, revert the operator UI state and surface a reject advisory.
2. **`$--HTC` (B1)** — Relay to autopilot without modification. On HBT loss, cease forwarding.
3. **`$PAVKHOMS` (C1)** — Extract all four mode fields; treat as authoritative steady-state for TC / CAGA / Radar / PTZ mode displays.
4. **`track_control_info` (C2)** — Surface to the track-controller operational panel; null values must be rendered as "--" rather than zero.
5. **`nas_indication_schema` (C4)** — Cache under `(version_major, version_minor, version_hotfix)`. Discard cached schema when a higher version is observed.
6. **`$PAVKINDD` (C5)** — Merge each `(mode, indication_id, state)` tuple into the operator indication panel, resolving `title` / `description` from the cached schema.

### 6.3 State & Alerting

| Event                                   | VA Action                                                            |
| --------------------------------------- | -------------------------------------------------------------------- |
| CAGM command issued                     | Start 1.0 s echo timer; mark CAGA toggle pending                     |
| CAGM echo `result = A` within timeout   | Clear pending; reset retry counter                                   |
| CAGM echo `result = R`                  | Revert CAGA UI; surface reject advisory                              |
| CAGM echo not received within 1.0 s     | Retransmit (up to 3 times)                                           |
| CAGM 3 retries exhausted                | Declare command failed; alert operator                               |
| 5 s silence on `$PAVKHOMS`              | Mark all four modes unknown; chain into LINK LOST                    |
| `$PAVKHOMS` mode transition (any field) | Update corresponding UI badge; log event                             |
| 5 s silence on `track_control_info`     | Mark track-controller state stale                                    |
| 5 s silence on `$PAVKINDD`              | Hold last indication states; chain into LINK LOST if HBT also silent |
| Indication schema version mismatch      | Re-send `$PKSOINDR` with empty version fields                        |
| HBT link lost                           | Cease HTC relay to autopilot immediately (failsafe)                  |

### 6.4 Authority Rules

- **`$PAVKHOMS` is the single source of truth** for steady-state mode. Command echoes (`$PAVKCAGM`, `$PAVKPTZM`, `$PAVKRDRM`) are transactional receipts, not steady-state.
- **Indication state (`$PAVKINDD`) takes precedence over `$PAVKHOMS`** when determining operator-visible health status. A mode reported `A` (Activate) in HOMS but `F` (Fault) in Indication must be displayed as faulted.
- **`$--HTC` is relay-only.** VA must not originate, interpolate, cache, or replay HTC values.
- **Schema must be cached before snapshot interpretation.** If `$PAVKINDD` arrives before the schema is cached, buffer the snapshot and replay it once the schema response arrives.
